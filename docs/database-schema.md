# Database Schema & Data Model

The baseline uses an **H2 in-memory database** (`jdbc:h2:mem:demo`, `application.yml` line 3) with a single entity: `User`. The schema is auto-generated by Hibernate at application startup because `spring.jpa.hibernate.ddl-auto: update` is set (`application.yml` line 8).

## DDL for `users`

Hibernate generates the table from the `@Entity` + `@Table(name = "users")` annotations on `User.java` lines 21-22. The DDL that Hibernate emits on H2 is effectively:

```sql
CREATE TABLE users (
    id    BIGINT       NOT NULL AUTO_INCREMENT PRIMARY KEY,
    name  VARCHAR(80)  NOT NULL,
    email VARCHAR(255) NOT NULL
);
```

Column derivation:

| Column | Java field | Source | Notes |
| :--- | :--- | :--- | :--- |
| `id` | `Long id` | `User.java` lines 31-33 | `@Id` + `@GeneratedValue(strategy = GenerationType.IDENTITY)` — Hibernate delegates to the database's auto-increment. |
| `name` | `String name` | `User.java` lines 35-37 | `@NotBlank` + `@Size(min = 2, max = 80)`. Hibernate honors the `@Size` max and emits `VARCHAR(80)`. |
| `email` | `String email` | `User.java` lines 39-41 | `@NotBlank` + `@Email`. No `@Size` constraint — Hibernate defaults to `VARCHAR(255)`. |

---

## Identity generation strategy

The entity uses `GenerationType.IDENTITY` (`User.java` line 32). On H2 this maps to `AUTO_INCREMENT`, and Hibernate issues a `SELECT SCOPE_IDENTITY()` after each `INSERT` to retrieve the generated key.

Implications:

- Writes are **not batchable** — Hibernate must flush per row to get the id back. For the demo this is a non-issue; at scale, consider `GenerationType.SEQUENCE` or `GenerationType.TABLE` on databases that support sequences.
- Ids are only guaranteed sequential within a single JVM. Restarting the app resets the counter because `jdbc:h2:mem:demo` is wiped on JVM exit (the `DB_CLOSE_DELAY=-1` keeps it alive **during** the JVM lifetime, but not across restarts).

## `equals` / `hashCode` semantics

`User` declares custom `equals`/`hashCode` on `User.java` lines 43-54:

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User)) return false;
    User other = (User) o;
    return id != null && Objects.equals(id, other.id);
}

@Override
public int hashCode() {
    return Objects.hash(id);
}
```

What this means in practice:

1. **Two transient entities are never equal.** Before the first `save(...)`, the id is `null`, so `id != null && Objects.equals(id, other.id)` is `false` even for `this == o`? No — the short-circuit `if (this == o) return true;` guarantees reference equality still works. Two *different* unsaved instances, however, compare as unequal.
2. **Hash code flips after persisting.** `Objects.hash(null)` is not the same as `Objects.hash(1L)`, so an instance that was inserted into a `HashSet` before being saved and then saved will become unretrievable — the standard JPA pitfall.
3. **Safe to use in `HashSet`/`HashMap` only after `save()`.** Add a `User` to a hash-based collection only after the repository has assigned it an id, or use a business-key-based implementation.

The alternative patterns you'll see in the wild:

- **Natural key** — override based on a unique business identifier such as `email`. Stable across sessions, but requires that the field is truly unique and immutable.
- **UUID primary keys** — assign `UUID.randomUUID()` in `@PrePersist` or directly in the constructor, then `equals`/`hashCode` on the UUID is stable even before insert.

See the **Known Limitations & Technical Debt** page for the broader trade-off.

## Enabling the H2 console

The baseline does **not** expose the H2 web console. To inspect the in-memory database during local development, add the following to `application.yml` (do not commit these to `main`):

```yaml
spring:
  h2:
    console:
      enabled: true
      path: /h2
```

Then permit the console URL in the security config. In `SecurityConfig.java` (baseline), inside `configure(HttpSecurity http)`, add a matcher **before** `anyRequest().authenticated()`:

```java
http
    .csrf().disable()
    .authorizeRequests()
        .antMatchers("/api/greet", "/h2/**").permitAll()
        .anyRequest().authenticated()
    .and()
    .httpBasic()
    .and()
    .headers().frameOptions().sameOrigin();   // allow the console's iframe UI
```

Then open http://localhost:8080/h2, use the JDBC URL `jdbc:h2:mem:demo`, username `sa`, empty password.

On the Spring Boot 3 target, the same configuration is expressed with `requestMatchers(...)` and the lambda DSL. See the **Upgrade Guide** for the rewritten security config.
